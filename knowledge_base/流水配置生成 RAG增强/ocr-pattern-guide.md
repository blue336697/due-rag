# OCR 模式识别指南

本文档帮助 LLM 从 PaddleOCR-VL 产出的结构化数据中识别表格结构、推断配置参数。供首次生成配置时注入 prompt。

---

## 一、判断 merge_mode（合并模式）

| OCR 表格特征 | merge_mode | 说明 |
|-------------|-----------|------|
| 每行是一条完整交易，各项字段（日期、金额、余额、对手方等）在同一行内 | `standard` | 绝大多数银行流水 |
| 每笔交易占两行，第一行和第二行交错排列，如第一笔的对手方在第二行、金额在第一行 | `interleaved` | 微信支付流水典型特征 |

### 判断线索

- **standard**：HTML 表格每行 (`<tr>`) 包含 5-10 个 `<td>`，列数一致，每行有日期或金额
- **interleaved**：HTML 表格的 `<tr>` 中列数偏少（3-5 列），相邻行内容互补，且缺少明显的"一行一条交易"结构

---

## 二、判断 amount_type（金额类型）

| OCR 表格特征 | amount_type | income_expense | 示例银行 |
|-------------|------------|----------------|---------|
| 表格中只有一列金额列，金额带 `+`/`-` 号或正负号 | `A` | `amount_sign` | 招商银行、中信银行、邮储银行 |
| 表格中有独立的两列——"收入"列和"支出"列 | `B` | `separate_columns` | 光大银行 |
| 有独立收支两列，且每行有"收入"/"支出"或"收"/"支"文字标签 | `B_with_label` | `separate_columns` 或 `direction_column` | 部分银行老版式 |

### 判断线索

- 表头含"收入金额"/"存入金额"/"贷方金额"其中之一 → 这是收入列
- 表头含"支出金额"/"支取金额"/"借方金额"其中之一 → 这是支出列
- 同时出现上述两类 → `B` 或 `B_with_label`
- 只出现"交易金额"/"金额"/"发生额" → `A`

---

## 三、判断 direction（收支方向）

| OCR 表格特征 | 配置方式 |
|-------------|---------|
| 金额列数值有 `+`、`-` 前缀或正负号 | `determineIncomeExpenseByAmount: true` |
| 有独立的一列标注"收"/"支"或"收入"/"支出" | `direction_column` 模式: `{"mode":"direction_column","column":"列名"}` |
| 有独立收支两列 | `separate_columns` 模式 |

### 常见中文方向标签

| OCR 文本 | 对应方向 |
|---------|---------|
| 收入、存入、贷方、贷、转入、汇入 | income |
| 支出、支取、借方、借、转出、汇出 | expense |
| 收、支、其他 | income / expense / 不计 |

## 四、日期格式识别

| OCR 日期样例 | format |
|-------------|--------|
| 2024-01-15 | `YYYY-MM-DD` |
| 2024/01/15 | `YYYY/MM/DD` |
| 2024.01.15 | `YYYY.MM.DD` |
| 15/01/2024（日/月/年，较少见） | `DD/MM/YYYY` |
| 2024-01-15 14:30:00 | `YYYY-MM-DD HH:mm:ss` |
| 2024年01月15日 | `YYYY年MM月DD日` |
| 20240115（8位数字） | 用 `date_format` + `format_param: "yyyyMMdd"` |

### 判断线索

- 日期通常在最左侧或最右侧列
- 日期和时间的分离：如果日期和时间分两列（如"交易日期"+"交易时间"），在 `date.columns` 中填两个列名，用 `combine_fields` 合并
- 8 位纯数字日期通常需在 `format_param` 中指定格式

## 五、常见中文列名 → 标准字段映射

以下表格列出银行流水 OCR 中常见的中文列名及其对应的标准字段。当 LLM 看到这些列名时，应映射到对应标准字段。

### 日期类

| OCR 常见列名 | 标准字段 |
|-------------|---------|
| 交易日期、记账日期、入账日期、起息日、业务日期 | `date` |
| 交易时间、记账时间、业务时间 | 配合日期列合并 |
| 交易日期时间、交易时间戳 | `date`（含时分秒） |

### 金额类

| OCR 常见列名 | 标准字段 |
|-------------|---------|
| 交易金额、金额、发生额、金额(元)、交易金额(元) | `amount`（amount_type=A） |
| 收入金额、存入金额、贷方金额、转入金额 | `income`（amount_type=B） |
| 支出金额、支取金额、借方金额、转出金额 | `expense`（amount_type=B） |
| 币种、币别、货币 | `currency` |

### 余额类

| OCR 常见列名 | 标准字段 |
|-------------|---------|
| 账户余额、余额、联机余额、可用余额、交易余额、本次余额 | `balance` |

### 对手方类

| OCR 常见列名 | 标准字段 |
|-------------|---------|
| 交易对手方、对方户名、对手信息、交易对方、对方名称、对方账户名 | `counterparty`（个人） |
| 对方名称、交易对手名称、对手户名 | `counterparty_name`（对公） |
| 对方账号、对手账号、交易对手账号 | `counterparty_account`（对公） |
| 对方开户行、对手开户行、对方银行 | `counterparty_bank`（对公） |

### 摘要/备注类

| OCR 常见列名 | 标准字段 |
|-------------|---------|
| 交易摘要、备注、附言、用途、交易附言、业务摘要、摘要 | `summary` 或 `trade_remark` |

### 交易类型/方式类

| OCR 常见列名 | 标准字段 |
|-------------|---------|
| 交易类型、交易名称、业务类型 | `trade_type` |
| 交易方式、支付方式、渠道、收付款方式、交易渠道 | `payment_method` |
| 交易单号、流水号、业务编号 | `reference_no` |

### 方向类

| OCR 常见列名 | 标准字段 |
|-------------|---------|
| 收/支、收支、方向、借贷标志、借贷方向 | `direction` |

## 六、表头行识别线索

### by_keyword 模式的关键词选择

选择哪个列名作为 `header_keyword`：
1. **首选日期列名**（"交易日期"/"交易时间"/"记账日期"）：日期列几乎总是存在，辨识度高
2. **次选金额列名**（"交易金额"/"金额"）：金额列也很常见，但可能出现在数据行文本中
3. **避免通用词**（"序号"/"备注"/"对方"）：这些词可能在数据行中多次出现导致误判

### 双语表头识别

如果表头含中英文双语（如 "Trans Date 交易日期"、"Dc Flg 借贷标志"），需设置 `has_english_header: true`。

常见英文表头列名：
- Trans Date / Value Date → 日期
- Debit / Credit → 借/贷
- Balance → 余额
- Transaction Amount → 金额
- Transaction Description / Narrative → 摘要

## 七、元数据提取线索

从 OCR 的 `text` 块或文档头部提取以下元数据：

| 元数据字段 | OCR 中的常见模式 | extract_method |
|-----------|----------------|---------------|
| 账户名/户名 | "户名：张三"、"账户名称：张三" | `regex`，pattern: `户名[：:]\\s*(.+)` |
| 账号/卡号 | "账号：6222****1234"、"卡号：6222****1234" | `regex`，pattern: `(?:账号|卡号)[：:]\\s*([0-9*]+)` |
| 起始日期 | "起始日期：2024-01-01"、"账单周期：2024.01.01-2024.12.31" | `regex` |
| 结束日期 | 同起始日期，取范围的结束部分 | `regex` |
| 币种 | "币种：人民币"、"CNY" | `keyword` 或 `direct: "CNY"` |
| 银行名称 | "招商银行"、"中国工商银行" 在文档标题中 | `keyword` |

## 八、multiline 判断线索

### 何时需要 multiline

当以下情况出现时，需要在 `parse_config.multiline` 中配置：

1. **对手方名称跨行**：一个对手方名称在表格中占两行
2. **摘要/备注跨行**：摘要文本很长，分行显示
3. **跨页断行**：一条交易记录的首行在页末，续行在下一页开头

### 配置方式

```json
"multiline": {"fields": ["counterparty", "summary"], "strategy": "join"}
```

- `strategy: "join"`：多行文本直接拼接
- `strategy: "split"`：多行文本空格分隔

### 判断线索

- 查看 HTML 表格中是否存在 `rowspan` 或合并单元格
- 查看是否有某些行的列数明显少于表头列数（可能是跨行数据的续行）
- 标准银行流水通常 `strategy="join"`

## 九、特殊情况处理

### 对公 vs 个人判断

| OCR 特征 | statement_type |
|----------|---------------|
| 文档标题含"个人"、"借记卡"、"储蓄卡"、"个人活期" | `personal` |
| 文档标题含"对公"、"公司"、"企业"、"单位"、"结算账户" | `corporate` |
| 金额单位为"万元"而非"元" | 大概率对公 |
| 交易金额普遍在 10 万以上 | 大概率对公 |
| 对手方中出现公司全称（含"有限公司"/"股份有限公司"） | 大概率对公 |
| 无法判断 | `unknown` |

### 个人流水中出现对公特征列

如果 `statement_type=personal` 但表格中出现了"对方账号"/"对方开户行"列：
- 仍然可以映射这些列（作为额外信息），但不设为 core 字段
- `counterparty` 设为主对手方列，`counterparty_account` 和 `counterparty_bank` 可通过 `model_extra` 放入 merge_config

### 对公流水中列数特别多

- 对公流水列数可能多达 15-20 列
- 优先覆盖核心字段（date/amount/counterparty_name/balance）
- 其余列通过 `model_extra` 放入 merge_config 或忽略

## 十、常见银行表格特征速查

| 银行 | merge_mode | amount_type | 典型表头关键字 | 备注 |
|------|-----------|------------|--------------|------|
| 中国银行 | standard | A | 记账日期 | 日期+时间分两列，需 combine_fields |
| 招商银行 | standard | A | 交易日期 | 金额带正负号 |
| 中信银行 | standard | A | 交易日期 | 扫描件 |
| 邮储银行 | standard | A | 交易日期 | 文本型 PDF |
| 光大银行 | standard | B | 记账日期 | 收入/支出分列 |
| 工商银行 | standard | A | 交易日期 | 字段齐全 |
| 交通银行 | standard | A | 交易日期 | 可能有英文列名 |
| 微信支付 | interleaved | B_with_label | 交易时间 | 交错双行，有收/支标签列 |
| 支付宝 | standard | A | 交易时间 | 类似微信但 standard 模式 |
