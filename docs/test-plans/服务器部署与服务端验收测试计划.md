# RAG 服务服务器部署与服务端验收测试计划

## 1. 目的

本计划交给服务器上的 AI 执行，用于验证 RAG 服务在服务器环境中的：

- 代码版本和部署配置可追溯；
- Qdrant、Redis、Embedding、Reranker、LLM 和公司 OCR 可用；
- 人工知识库只读、系统发布目录可写；
- 上传、解析、预览、发布、Reindex、搜索、删除形成完整闭环；
- 容器重启后任务、文档和索引可以恢复；
- 失败场景不会破坏当前在线索引。

## 2. 执行约束

1. 只能在测试或预发布服务器执行。不得直接使用生产 Qdrant、Redis、索引目录和知识库目录。
2. 不得删除既有数据，不得执行 `git reset --hard`、递归删除知识库或索引目录。
3. 日志和报告中不得输出 `RAG_SERVICE_API_KEY`、`LLM_API_KEY` 等密钥。
4. 每一步记录命令退出码、HTTP 状态码、job ID 和关键响应；失败后停止依赖该步骤的后续用例。
5. 当前待部署代码必须先提交并推送。服务器必须记录实际测试的 commit ID。

## 3. 测试变量

```bash
export RAG_BASE_URL="http://127.0.0.1:8020"
export RAG_DOMAIN="bank_stmt"
export TEST_RUN_ID="$(date +%Y%m%d%H%M%S)"
export TEST_TOKEN="RAG_SERVER_E2E_${TEST_RUN_ID}"
export COMPOSE_FILE="docker/docker-compose.yml"
```

`RAG_SERVICE_API_KEY` 和 `LLM_API_KEY` 应从服务器安全配置注入，不在命令或报告中打印。

## 4. 前置条件

- Linux 服务器可以访问公司 OCR：`http://192.168.160.88:7557`。
- 服务器已经安装 Docker 和 Docker Compose。
- 模型已放到 `/opt/uboss/haojie.liu/rag-data/models/huggingface/`。
- 测试知识库至少包含一份有效 Markdown。
- 已准备一份非敏感、1～2 页的扫描 PDF，建议包含正文和简单表格。
- 本机安装 `curl`、`jq`、`sha256sum`。

## 5. 通过标准

以下条件必须全部满足：

- 17 个自动化测试通过；
- Docker Compose 配置检查和镜像构建成功；
- `/health` 返回 `200`；完成初始 Reindex 后 `/ready` 返回 `200` 且 `bank_stmt.status=ok`；
- `vector_available=true`、`rerank_available=true`；
- 人工知识库挂载为只读，managed 目录挂载为可写；
- Markdown 和扫描 PDF 都能完成摄取；
- 发布后能检索到唯一测试标识，来源以 `_managed/` 开头；
- 容器重启后仍能检索；
- 删除并 Reindex 后不能再检索到测试标识；
- 整个过程中无未解释的 5xx、任务失败或索引回退。

## 6. 测试用例

### S01：代码版本和工作区检查

```bash
git rev-parse HEAD
git status --short
git log -1 --format='%H %ad %s' --date=iso
```

预期：

- 记录 commit ID；
- 工作区为空；
- commit 与本地计划部署版本一致。

### S02：宿主机目录准备

```bash
mkdir -p /opt/uboss/haojie.liu/rag-data/knowledge_base
mkdir -p /opt/uboss/haojie.liu/rag-data/managed_knowledge
mkdir -p /opt/uboss/haojie.liu/rag-data/indexes
mkdir -p /opt/uboss/haojie.liu/rag-data/ingestion
mkdir -p /opt/uboss/haojie.liu/rag-data/logs
mkdir -p /opt/uboss/haojie.liu/rag-data/qdrant
mkdir -p /opt/uboss/haojie.liu/rag-data/redis
```

确认模型：

```bash
test -d /opt/uboss/haojie.liu/rag-data/models/huggingface/BAAI/bge-small-zh-v1.5
test -d /opt/uboss/haojie.liu/rag-data/models/huggingface/BAAI/bge-reranker-base
```

预期：所有命令退出码为 `0`。

### S03：静态检查和自动化测试

```bash
python -m py_compile service/main.py service/core/config.py service/retrieval/service.py
python -m unittest tests.test_ingestion tests.test_reindex_hardening tests.test_security -q
docker compose -f "$COMPOSE_FILE" config --quiet
```

预期：17 个测试全部通过，Compose 配置合法。

### S04：构建并启动服务

```bash
docker compose -f "$COMPOSE_FILE" up -d --build
docker compose -f "$COMPOSE_FILE" ps
docker logs rag-service --tail 200
```

预期：

- `rag-qdrant`、`rag-redis`、`rag-service` 均为运行状态；
- 日志中没有模型加载失败、权限错误和持续重启；
- 不允许因为 Qdrant 不可用而静默切换到内存索引。

### S05：挂载权限验证

```bash
docker inspect rag-service --format '{{json .Mounts}}' | jq .
docker exec rag-service sh -c 'test -r /app/knowledge_base'
docker exec rag-service sh -c 'test -w /managed-knowledge'
```

预期：

- `/app/knowledge_base` 的 `RW` 为 `false`；
- `/managed-knowledge` 的 `RW` 为 `true`；
- `/indexes` 和 `/ingestion` 可写。

在后续测试前记录人工知识库摘要：

```bash
find /opt/uboss/haojie.liu/rag-data/knowledge_base -type f -print0 \
  | sort -z | xargs -0 -r sha256sum | sha256sum \
  > "/tmp/knowledge-before-${TEST_RUN_ID}.sha256"
```

### S06：依赖服务和 OCR 连通性

```bash
curl -fsS http://127.0.0.1:6333/collections >/dev/null
docker exec rag-redis redis-cli ping
docker exec rag-service python -c "import httpx; r=httpx.get('http://192.168.160.88:7557/openapi.json', timeout=5); print(r.status_code); assert r.status_code == 200; assert '/api/v1/file_parse' in r.text"
```

预期：Qdrant 可访问、Redis 返回 `PONG`、OCR OpenAPI 返回 `200`。

### S07：健康检查和鉴权

```bash
curl -i "$RAG_BASE_URL/health"
curl -i "$RAG_BASE_URL/ready"
curl -i -X POST "$RAG_BASE_URL/rag/search" \
  -H 'Content-Type: application/json' \
  -d '{"query":"unauthorized","domain":"bank_stmt"}'
```

预期：

- `/health` 返回 `200`；
- 初次未索引时 `/ready` 允许返回 `503 unindexed`；
- 未携带 API Key 的搜索返回 `401`。

### S08：初始 Reindex

```bash
REINDEX_RESPONSE="$(curl -fsS -X POST "$RAG_BASE_URL/admin/reindex" \
  -H "X-API-Key: $RAG_SERVICE_API_KEY" \
  -H 'Content-Type: application/json' \
  -d '{"domain":"bank_stmt","force":true}')"
REINDEX_JOB_ID="$(printf '%s' "$REINDEX_RESPONSE" | jq -r .job_id)"
test -n "$REINDEX_JOB_ID"
```

每 2 秒轮询：

```bash
curl -fsS -H "X-API-Key: $RAG_SERVICE_API_KEY" \
  "$RAG_BASE_URL/admin/reindex/$REINDEX_JOB_ID" | jq .
```

预期：最终 `status=completed`，`parent_count>0`、`child_count>0`。随后：

```bash
curl -fsS "$RAG_BASE_URL/ready" | jq .
```

必须为 `200`，且 `bank_stmt.status=ok`。

### S09：Markdown 摄取、发布和检索闭环

创建测试文件：

```bash
printf '# 服务器部署验收材料\n\n## 唯一规则\n\n验证码 %s 表示服务器知识库发布链路正常。\n' \
  "$TEST_TOKEN" > "/tmp/rag-server-${TEST_RUN_ID}.md"
```

上传：

```bash
UPLOAD_RESPONSE="$(curl -fsS -X POST "$RAG_BASE_URL/admin/documents" \
  -H "X-API-Key: $RAG_SERVICE_API_KEY" \
  -F "file=@/tmp/rag-server-${TEST_RUN_ID}.md" \
  -F "domain=$RAG_DOMAIN" \
  -F 'metadata={"source":"server-e2e"}')"
DOCUMENT_ID="$(printf '%s' "$UPLOAD_RESPONSE" | jq -r .document_id)"
INGEST_JOB_ID="$(printf '%s' "$UPLOAD_RESPONSE" | jq -r .job_id)"
```

轮询 `/admin/ingestions/$INGEST_JOB_ID`，预期最终为 `ready`。然后：

```bash
curl -fsS -H "X-API-Key: $RAG_SERVICE_API_KEY" \
  "$RAG_BASE_URL/admin/documents/$DOCUMENT_ID/preview" | jq .
```

预期 preview 包含 `$TEST_TOKEN` 且 `quality.passed=true`。

发布：

```bash
PUBLISH_RESPONSE="$(curl -fsS -X POST \
  -H "X-API-Key: $RAG_SERVICE_API_KEY" \
  "$RAG_BASE_URL/admin/documents/$DOCUMENT_ID/publish")"
PUBLISH_REINDEX_JOB_ID="$(printf '%s' "$PUBLISH_RESPONSE" | jq -r .reindex_job_id)"
```

轮询 Reindex 到 `completed`，并确认文件存在：

```bash
test -f "/opt/uboss/haojie.liu/rag-data/managed_knowledge/bank_stmt/${DOCUMENT_ID}.md"
```

搜索：

```bash
curl -fsS -X POST "$RAG_BASE_URL/rag/search" \
  -H "X-API-Key: $RAG_SERVICE_API_KEY" \
  -H 'Content-Type: application/json' \
  -d "{\"query\":\"$TEST_TOKEN\",\"domain\":\"bank_stmt\",\"top_k\":5,\"mode\":\"hybrid\"}" \
  | tee "/tmp/search-${TEST_RUN_ID}.json" | jq .
```

预期：结果包含 `$TEST_TOKEN`，且 `source` 为 `_managed/${DOCUMENT_ID}.md`。

### S10：公司 OCR 摄取闭环

使用预先准备的非敏感扫描 PDF：

```bash
export OCR_TEST_PDF="/path/to/non-sensitive-scan.pdf"
test -f "$OCR_TEST_PDF"
```

按照 S09 的上传和轮询流程上传 PDF。预期：

- ingestion 最终为 `ready` 或 `awaiting_review`，不得为 `failed`；
- preview 的 `metadata.ocr_used=true`；
- `metadata.ocr_model=br-ocr-v1`；
- `metadata.ocr_provider=company_flow`；
- `normalized_markdown` 非空；
- 原 PDF 有表格时，规范 Markdown 中存在表格结构。

如果状态为 `awaiting_review`，记录全部 warnings。只有人工确认内容正确后才允许使用 `force_quality=true` 发布。

### S11：重启和持久化

在删除 S09 文档之前执行：

```bash
docker restart rag-service
```

等待健康检查恢复，再搜索 `$TEST_TOKEN`。预期：

- `/health` 和 `/ready` 恢复；
- Manifest generation 没有倒退；
- `$TEST_TOKEN` 仍可检索；
- ingestion 文档和任务记录仍存在。

### S12：删除和索引下线

```bash
DELETE_RESPONSE="$(curl -fsS -X DELETE \
  -H "X-API-Key: $RAG_SERVICE_API_KEY" \
  "$RAG_BASE_URL/admin/documents/$DOCUMENT_ID")"
DELETE_REINDEX_JOB_ID="$(printf '%s' "$DELETE_RESPONSE" | jq -r .reindex_job_id)"
```

轮询 Reindex 到 `completed`。预期：

- managed Markdown 已删除；
- Manifest generation 增加；
- 再次搜索 `$TEST_TOKEN` 不得命中；
- 其他人工知识仍可正常检索。

### S13：人工知识库未被修改

```bash
find /opt/uboss/haojie.liu/rag-data/knowledge_base -type f -print0 \
  | sort -z | xargs -0 -r sha256sum | sha256sum \
  > "/tmp/knowledge-after-${TEST_RUN_ID}.sha256"
diff "/tmp/knowledge-before-${TEST_RUN_ID}.sha256" \
     "/tmp/knowledge-after-${TEST_RUN_ID}.sha256"
```

预期：`diff` 无输出。

### S14：错误场景

至少验证：

| 场景 | 预期 |
|---|---|
| 不携带 API Key | `401` |
| 错误 API Key | `401` |
| 未知 domain | `400` 或 `404` |
| 不支持的扩展名 | `400` |
| metadata 不是 JSON 对象 | `400` |
| `top_k=0` | `422` |
| 不存在的 document ID | `404` |
| 不存在的 reindex job | `status=not_found` |

错误场景不得造成容器重启、当前 Manifest 损坏或已发布索引不可用。

## 7. 清理

- 删除本计划上传的所有测试文档，并等待对应 Reindex 完成；
- 删除 `/tmp/rag-server-*`、临时响应和摘要文件；
- 不删除 Qdrant、Redis、ingestion 或 indexes 整体目录；
- 保留测试日志、job ID 和最终报告。
