基础服务、鉴权、检索、远程上传、发布、原子 Reindex、检索命中、删除下线、连续请求和并发请求均可用。本地远程调用主链路通过。

本轮发现 3 个需要修复的接口/数据完整性问题：

1. keyword/vector 模式没有遵守请求的 `top_k`；
2. Answer 的 LLM 调用失败后静默返回原始检索片段，HTTP 仍为 200；
3. 由 keyword 召回进入 hybrid 的部分结果丢失 `chunk_id` 和 `content_hash`，导致引用不可完整追溯。

此外，5 路并发虽然全部成功，但单请求延迟上升到约 4.6～4.8 秒；浏览器 CORS 预检当前不支持。

## 9. 问题清单

### P1：Answer 静默降级为原始片段

现象：`/rag/answer` 返回 HTTP 200，但 `answer` 是 `--- 片段 [1] ---` 开头的检索上下文拼接，而非 LLM 生成的自然语言答案。

代码依据：`service/models/answer_llm.py` 捕获所有异常后直接返回 `chunks_text`。服务端需查看 `docker logs rag-service` 中的“LLM 生成回答失败”日志确认实际异常。

建议：

- 修复 LLM base URL、模型名、认证或依赖问题；
- 响应中增加 `answer_mode=llm|fallback` 和降级原因；
- 不要把原始片段拼接伪装成正常 LLM 答案。

### P1：部分 Citation 缺少追溯字段

现象：Answer 的前 3 条 citation 中 `chunk_id` 和 `content_hash` 为空，后 2 条正常。

代码原因：`service/storage/bm25.py::keyword_recall()` 构造新字典时只保留 `source/heading/content/keyword_score`，丢弃 chunk 标识和 metadata；hybrid 去重优先保留 keyword 版本，导致引用字段无法恢复。

建议：keyword recall 返回原 chunk 的 `chunk_id`、`parent_id`、`heading_path`、`content_hash` 和 metadata，并增加完整性测试。

### P2：Keyword/Vector 不遵守请求 top_k

现象：请求 `top_k=5`，keyword 返回配置召回数 20，vector 返回配置召回数 40。

代码原因：`service/retrieval/service.py::_search_hybrid_advanced()` 的单模式分支直接使用 `top_k_keyword` 和 `top_k_vector`，返回前没有按请求 `top_k` 截断。

建议：召回阶段可以继续使用 20/40，但 API 返回前必须执行 `results[:top_k]`。

### P2：并发延迟近似线性增长

现象：单请求约 0.85～0.93s，5 路并发每路约 4.6～4.8s。

建议：根据真实调用量进行 10/20 路压测；评估多 worker、多模型副本、GPU 或推理队列。

### P3：浏览器直连不支持 CORS

现象：OPTIONS `/rag/search` 返回 405，无 `Access-Control-Allow-Origin`。

影响：due-agent 后端、Python、curl 和服务间调用不受影响；浏览器前端直连会被阻止。

建议：优先通过同源后端代理；如必须直连，增加严格来源白名单。
